Define 'Loose Coupling'.
Loose coupling means that classes or components in a program are as independent from each other as possible, meaning a change in one class should have little to no impact on another class. When two classes are tightly coupled it means one class directly depends on the concrete implementation of another class, so if you change something in one you are forced to change the other as well which makes the code rigid and hard to maintain. Loose coupling solves this by making classes depend on abstractions like interfaces or abstract classes instead of concrete implementations. For example instead of writing `SBI s = new SBI()` everywhere in your code, if you write `Bank b = new SBI()` and work with the `Bank` reference, the rest of your code does not care whether the actual object is SBI or HDFC or any other bank, you can swap the implementation without touching anything else. This is exactly why interfaces are so heavily used in real world Java projects, they act as a contract between classes so that the classes interact through the interface and do not need to know anything about each other's internal implementation. Loose coupling makes code easier to maintain, easier to test, easier to extend, and easier to scale because individual components can be changed or replaced independently without breaking the whole system. So basically loose coupling is about minimizing direct dependencies between classes by programming to abstractions rather than concrete implementations, and it is one of the most important principles of good object oriented design.
